## 1. Общее описание

**REDIPS Blockchain** — это учебная реализация блокчейна с доказательством работы (Proof of Work) на языке Rust. Система представляет собой REST API-сервер, который предоставляет полный цикл работы с блокчейном: создание кошельков, отправку транзакций, майнинг блоков и просмотр состояния цепочки.

**Основные характеристики:**
- Алгоритм консенсуса: **Proof of Work (PoW)**
- Хеш-функция блока: **двойной SHA-256** (как в Bitcoin)
- Адресация кошельков: **SHA-256 → RIPEMD-160** (как в Bitcoin)
- Криптография транзакций: **ECDSA на кривой secp256k1**
- Генерация кошельков: **BIP-39 мнемоника** (12 слов)
- Хранилище данных: **sled** (встроенная key-value БД)
- Нативный токен: **REDIPS**
- Максимальная эмиссия: **100 000 000 REDIPS**
- Начальная награда за блок: **1000 REDIPS**
- Халвинг каждые: **60 480 блоков**

## 3. Модули и их назначение

### `types.rs`
Содержит все публичные структуры данных:

| Структура | Назначение |
|---|---|
| `Transaction` | Единица перевода средств между адресами |
| `TransactionType` | Перечисление типов транзакций (`Transfer`) |
| `Block` | Блок в цепочке (содержит транзакции, хеши, nonce и т.д.) |
| `Wallet` | Адрес, публичный и приватный ключи пользователя |
| `BlockchainStats` | Статистика сети (блоки, supply, сложность, hashrate) |

### `crypto.rs`
Криптографические операции:
- Генерация и разбор BIP-39 мнемоники
- Деривация ключевой пары из seed-фразы
- Конвертация публичного ключа в адрес (SHA-256 → RIPEMD-160)
- ECDSA подпись и верификация транзакций
- Генерация случайных UUID-идентификаторов

### `block_utils.rs`
Утилиты уровня блока:
- Расчёт хеша блока (двойной SHA-256)
- Построение дерева Меркля
- Проверка соответствия хеша таргету сложности
- Цикл PoW-майнинга
- Корректировка сложности сети
- Расчёт награды за блок с учётом халвингов

### `blockchain.rs`
Бизнес-логика блокчейна:
- Создание и инициализация генезис-блока
- Добавление транзакций в mempool с валидацией
- Майнинг нового блока с защитой от параллельного запуска
- Применение транзакций блока к балансам
- Расчёт статистики сети

### `database.rs`
Слой хранения данных на sled:
- CRUD для блоков (ключ: `block:{index}`)
- Поиск транзакций (ключ: `tx:{id}`)
- Хранение балансов (ключ: `balance:{address}:{token_id}`)

### `api.rs`
REST HTTP-сервер на actix-web:
- Маршрутизация запросов
- Унифицированные ответы `ApiResponse<T>`
- Обработчики для кошельков, транзакций, майнинга и блокчейна

### `main.rs`
Точка входа:
- Загрузка конфигурации из `.env`
- Автогенерация мастер-кошелька при первом запуске
- Инициализация БД и блокчейна
- Запуск HTTP-сервера

## 4. Алгоритм работы системы

### Полный жизненный цикл

```
Запуск → Загрузка .env → Проверка мастер-кошелька
       → Открытие БД → Создание генезис-блока (если цепочка пуста)
       → Запуск HTTP-сервера
               │
    ┌──────────┴──────────┐
    │                     │
Пользователь          Майнер
создаёт кошелёк       вызывает POST /api/mine
отправляет TX         │
    │                  ↓
    ↓        blockchain.mine_block()
POST /api/transaction/send   │
    │                  ├─ validate_transaction()
    ↓                  ├─ Собрать TX из mempool
blockchain.add_transaction() ├─ Добавить coinbase TX (награда)
    │                  ├─ PoW цикл (nonce/extra_nonce)
    ├─ Валидация баланса    ├─ Проверить timestamp и хеш
    ├─ Добавить в mempool   ├─ process_block_transactions()
    └─ Сохранить в БД       └─ Сохранить блок в БД
```

### Добавление транзакции (`add_transaction`)

1. Вызвать `validate_transaction`:
   - Если отправитель не `SYSTEM` — проверить наличие подписи
   - Проверить баланс: `balance >= amount + fee`
2. Добавить транзакцию в `pending_transactions` (in-memory Mutex)
3. Сохранить транзакцию в БД для индексации по ID

### Майнинг блока (`mine_block`)

1. Проверить флаг `mining_in_progress` — в один момент времени только один майнинг
2. Получить последний блок из БД
3. Вычислить текущую сложность (`adjust_difficulty`)
4. Вычислить награду с учётом халвингов (`calculate_mining_reward`)
5. Проверить, что `circulating_supply + reward <= total_supply`
6. Взять до 100 транзакций из mempool
7. Создать coinbase-транзакцию (награда майнеру, `from = "SYSTEM"`)
8. Запустить PoW-цикл (`BlockUtils::mine_block`)
9. Проверить timestamp и хеш найденного блока
10. Применить транзакции блока к балансам (`process_block_transactions`)
11. Сохранить блок в БД

### Применение транзакций (`process_block_transactions`)

Для каждой транзакции в блоке:
1. Определить токен (из поля `data.tokenId`, по умолчанию `REDIPS`)
2. Если `from != "SYSTEM"` — списать `amount + fee` у отправителя
3. Зачислить `amount` получателю
4. Если `fee > 0` и `from != "SYSTEM"` — зачислить `fee` на мастер-кошелёк

## 5. Криптография

### BIP-39 мнемоника

Для генерации кошелька используется стандарт BIP-39:

```
128 бит случайной энтропии
        ↓
  BIP-39 encoding
        ↓
12 английских слов (мнемоника)
        ↓
  PBKDF2-HMAC-SHA512 (2048 итераций, salt = "")
        ↓
512-битный seed
        ↓
Первые 32 байта → SECRET KEY (secp256k1)
```

**Генерация (`generate_mnemonic`):**
- `rand::thread_rng()` заполняет 16-байтовый буфер (`OsRng`-based CSPRNG)
- `Mnemonic::from_entropy_in` кодирует энтропию в 12 слов

**Деривация ключей (`mnemonic_to_wallet`):**
- `mnemonic.to_seed("")` — deterministic seed через PBKDF2-HMAC-SHA512
- SHA-256 от первых 32 байт seed → приватный ключ secp256k1
- `PublicKey::from_secret_key` — публичный ключ из приватного

### Адрес кошелька

Алгоритм получения адреса из публичного ключа (аналог Bitcoin):

```
PublicKey (33 байта, compressed)
        ↓
     SHA-256
        ↓
   RIPEMD-160
        ↓
  "0x" + hex-encode
```

Пример: `0xa3f8b2c1d94e7f0123456789abcdef0123456789`

### ECDSA подпись транзакции

Используется кривая **secp256k1** — та же, что в Bitcoin и Ethereum:

```
sign_transaction(private_key, data):
  hash = SHA-256(data)
  message = secp256k1::Message::from_digest_slice(hash)
  signature = secp.sign_ecdsa(message, secret_key)
  return hex(signature.serialize_compact())  // 64 байта

verify_signature(public_key, signature, data):
  hash = SHA-256(data)
  secp.verify_ecdsa(message, signature, public_key) → bool
```

**Важно:** В текущей реализации API принимает подпись как строку в поле `signature` транзакции. Верификация на стороне сервера не выполняется автоматически — наличие непустой подписи является обязательным, а корректность проверяется на стороне клиента.

## 6. Механизм майнинга и PoW

### Хеш блока (двойной SHA-256)

```
data = JSON {
  index, timestamp, merkleRoot,
  previousHash, nonce, extraNonce,
  difficulty, miner
}

hash = SHA-256( SHA-256( data.to_string() ) )
```

Двойное хеширование применяется для устойчивости к **length-extension атакам**.

### Таргет сложности

Сложность `difficulty` задаёт количество ведущих нулевых **бит** в хеше:

```
difficulty = 20  →  первые 2 байта = 0x00, 0x00,
                    старшие 4 бита третьего байта = 0000
                 →  примерно 1 хеш из 2^20 (~1 млн) подходит
```

**Алгоритм проверки (`hash_meets_target`):**
```
full_bytes = difficulty / 8        // сколько байт должны быть нулями
remaining_bits = difficulty % 8    // оставшиеся биты в следующем байте

for i in 0..full_bytes:
    if hash[i] != 0 → неверно

mask = 0xFF >> remaining_bits
if hash[full_bytes] > mask → неверно
```

### PoW цикл

```
nonce = 0, extra_nonce = 0
loop:
    block.nonce = nonce
    block.extra_nonce = extra_nonce
    block.hash = SHA-256(SHA-256(block_data))
    if hash_meets_target(hash, difficulty):
        return block
    if nonce == u64::MAX:
        nonce = 0
        extra_nonce += 1    // расширяем пространство поиска до 2^128
    else:
        nonce += 1
```

**Параметры:**
- Начальная сложность: 20 бит
- Целевое время блока: 30 секунд
- Минимальный интервал между блоками: 5 секунд

## 7. Дерево Меркля

Дерево Меркля используется для эффективной проверки содержимого блока.

### Алгоритм построения

```
Транзакции: [tx1, tx2, tx3, tx4]

Уровень 0 (листья):
  H1 = SHA-256(SHA-256(tx1.id))
  H2 = SHA-256(SHA-256(tx2.id))
  H3 = SHA-256(SHA-256(tx3.id))
  H4 = SHA-256(SHA-256(tx4.id))

Уровень 1:
  H12 = SHA-256(H1 || H2)
  H34 = SHA-256(H3 || H4)

Корень:
  root = SHA-256(H12 || H34)
```

**Нечётное число транзакций:** последний элемент дублируется (как в Bitcoin):
```
[tx1, tx2, tx3]  →  [H1, H2, H3, H3]  →  [H12, H33]  →  root
```

**Пустой блок:** merkle_root = `"0000...0000"` (64 нуля).

Корень Меркля включается в данные для хеширования блока, что делает невозможным изменение содержимого транзакций без пересчёта хеша блока.

## 8. Корректировка сложности

Сложность пересчитывается каждые **144 блока**.

### Алгоритм

```
time_expected = 30_000 мс × 144 = 4 320 000 мс (72 минуты)
time_actual   = timestamp[last] - timestamp[last - 144]

ratio = time_expected / time_actual
bit_adjust = log2(ratio)

if bit_adjust >= 0:
    ideal = current + floor(bit_adjust)   // майнят быстрее → усложняем
else:
    ideal = current - floor(-bit_adjust)  // майнят медленнее → упрощаем

// Ограничиваем изменение: не более ±4 бит за раз
clamped = clamp(ideal, current - 4, current + 4)

// Нижний порог: не ниже начальной сложности (20)
new_difficulty = max(clamped, 20)
```

**Смысл log2:** если майнеры работают в 2× быстрее нормы → +1 бит, в 4× быстрее → +2 бита. Это логарифмическая шкала, аналогичная реальному биткоину.

## 9. Управление балансами

Балансы хранятся в sled по ключу `balance:{address}:{token_id}`.

### Халвинг награды

```
halvings = block_index / 60480
reward = 1000 / 2^halvings

Блок 0–60479:      reward = 1000 REDIPS
Блок 60480–120959: reward = 500 REDIPS
Блок 120960–181439: reward = 250 REDIPS
...
```

### Комиссии

Комиссия (`fee`) транзакции списывается у отправителя вместе с суммой и зачисляется на **мастер-кошелёк** (адрес из переменной `MASTER_WALLET_ADDRESS`).

## 11. База данных

Используется **sled** — встроенная, транзакционная key-value база данных для Rust (аналог LevelDB/RocksDB).

### Схема ключей

| Ключ | Значение | Тип |
|---|---|---|
| `block:{index}` | JSON-сериализованный `Block` | `Vec<u8>` |
| `tx:{id}` | JSON-сериализованная `Transaction` | `Vec<u8>` |
| `balance:{address}:{token_id}` | Сумма баланса как строка | `String` |

### Особенности

- `get_all_blocks()` сканирует все ключи с префиксом `block:` и сортирует по `index`
- `get_latest_block()` возвращает последний элемент отсортированного списка
- Все операции возвращают `Result<T>` через `anyhow`

## 13. Запуск проекта

### Требования
- Rust 1.75+
- Cargo

### Установка и запуск

```bash
# Клонировать репозиторий
git clone https://github.com/HRM6996/chain.git
cd chain

# Создать файл конфигурации
cp .env.example .env

# Запустить в режиме разработки
cargo run

# Или собрать оптимизированный бинарник
cargo build --release
./target/release/redips-blockchain
```

### Первый запуск

При первом запуске (пустая база данных):
1. Генерируется мастер-кошелёк — выводится адрес, приватный ключ и мнемоника
2. Данные кошелька сохраняются в `.env`
3. Создаётся генезис-блок (индекс 0, без транзакций)
4. Запускается HTTP-сервер

### Пример сессии через curl

```bash
# Создать кошелёк
curl -X POST http://localhost:3000/api/wallet/create

# Получить баланс
curl http://localhost:3000/api/wallet/0xabc.../balance

# Майнить блок
curl -X POST http://localhost:3000/api/mine \
  -H "Content-Type: application/json" \
  -d '{"minerAddress": "0xabc..."}'

# Статистика
curl http://localhost:3000/api/blockchain/stats
```
