❓ Зачем поле previous_hash в Block?
Это и есть «цепочка» в блокчейне. Каждый блок содержит хеш предыдущего. Если кто-то захочет изменить старый блок — изменится его хеш, и все последующие блоки станут невалидными. Это делает историю неизменяемой.
❓ Почему amount и fee хранятся как f64?
f64 — число с плавающей точкой. Удобно для вычислений наград и комиссий. В реальных production-блокчейнах (Bitcoin) используют целые числа (сатоши), чтобы избежать ошибок округления, но для учебного проекта f64 приемлем.
📄 src/crypto.rs — Криптография

Использует библиотеку secp256k1 — ту же эллиптическую кривую, что Bitcoin и Ethereum. Для мнемоник (seed-фраз) — стандарт BIP-39.

generate_mnemonic() — генерация seed-фразы

pub fn generate_mnemonic() -> String {
    let mut rng = rand::thread_rng();
    let mut entropy = [0u8; 16];  // 128 бит случайных данных
    rng.fill_bytes(&mut entropy);
    let mnemonic = Mnemonic::from_entropy_in(Language::English, &entropy).unwrap();
    mnemonic.to_string()  // 12 слов на английском
}

Что делает: генерирует 16 случайных байт, преобразует по стандарту BIP-39 в 12 понятных английских слов (например: witch collapse practice feed shame open). Эти слова — единственный способ восстановить кошелёк.

mnemonic_to_wallet() — создание кошелька из seed-фразы

pub fn mnemonic_to_wallet(mnemonic: &str) -> Result<Wallet> {
    let mnemonic = Mnemonic::parse_in(Language::English, mnemonic)?;
    let seed = mnemonic.to_seed("");      // 64 байта

    let secp = Secp256k1::new();
    let hash = Sha256::digest(&seed[..32]); // берём первые 32 байта → SHA256
    let secret_key = SecretKey::from_slice(&hash)?;       // приватный ключ
    let public_key = PublicKey::from_secret_key(&secp, &secret_key); // публичный ключ

    let address = Self::public_key_to_address(&public_key);
    Ok(Wallet { address, public_key: hex::encode(...), private_key: hex::encode(...) })
}

public_key_to_address() — вычисление адреса

pub fn public_key_to_address(public_key: &PublicKey) -> String {
    // SHA256(публичный_ключ) → RIPEMD160 → "0x" + hex
    let sha_hash = Sha256::digest(public_key.serialize());
    let address_hash = Ripemd160::digest(sha_hash);
    format!("0x{}", hex::encode(address_hash))
}

Двойное хеширование (SHA256 → RIPEMD160) — стандарт Bitcoin. Даёт адрес длиной 42 символа.

❓ Как работает асимметричная криптография в блокчейне?
У каждого пользователя — пара ключей: приватный (секретный, только у хозяина) и публичный (известен всем). Отправитель подписывает транзакцию приватным ключом. Все остальные проверяют подпись публичным ключом — и убеждаются, что именно этот человек дал разрешение на транзакцию, не зная его приватного ключа.
❓ Почему используется secp256k1, а не RSA?
secp256k1 — эллиптическая кривая. Обеспечивает такой же уровень криптостойкости, что и RSA-3072, но при значительно меньшем размере ключей. Это важно для блокчейна, где подписи хранятся в каждой транзакции.

Дерево Меркля (Merkle Tree)

pub fn calculate_merkle_root(transactions: &[Transaction]) -> String {
    // 1. Каждую транзакцию хешируем дважды: SHA256(SHA256(tx.id))
    let mut level = transactions.iter()
        .map(|tx| SHA256(SHA256(tx.id))).collect();

    // 2. Попарно объединяем хеши до одного корневого
    while level.len() > 1 {
        if level.len() % 2 != 0 { level.push(last.clone()); } // нечётное — дублируем
        level = level.chunks(2).map(|p| SHA256(p[0] + p[1])).collect();
    }
    hex::encode(level[0]) // корень дерева
}
Дерево Меркля — структура данных, позволяющая эффективно доказать, что транзакция входит в блок. Если изменить любую транзакцию — корень изменится, и блок станет невалидным. Нечётные листья дублируются (стандарт Bitcoin).

Хеширование блока (двойной SHA-256)

pub fn calculate_block_hash(block: &Block) -> String {
    let data = json!({
        "index": block.index, "timestamp": block.timestamp,
        "merkleRoot": block.merkle_root, "previousHash": block.previous_hash,
        "nonce": block.nonce, "extraNonce": block.extra_nonce,
        "difficulty": block.difficulty, "miner": block.miner,
    });
    let first  = SHA256(data.to_string());
    let second = SHA256(first);   // двойной SHA256 — защита от length-extension атак
    hex::encode(second)
}

Заметьте: транзакции не хешируются напрямую. В хеш блока входит только merkle_root — это и есть роль дерева Меркля.

Proof-of-Work (PoW) — майнинг

pub fn mine_block(...) -> Block {
    let mut nonce: u64 = 0;
    let mut extra_nonce: u64 = 0;
    loop {
        block.nonce = nonce;
        block.hash = calculate_block_hash(&block);

        if hash_meets_target(&block.hash, difficulty) {
            return block;  // найден! хеш удовлетворяет условию
        }

        if nonce == u64::MAX { nonce = 0; extra_nonce += 1; }
        else { nonce += 1; }   // пробуем следующий nonce
    }
}

Проверка difficulty (hash_meets_target)

pub fn hash_meets_target(hash_hex: &str, difficulty: u32) -> bool {
    let hash_bytes = hex::decode(hash_hex); // 32 байта
    let full_bytes = difficulty / 8;        // сколько байт = 0x00
    let remaining_bits = difficulty % 8;    // и сколько бит в следующем байте

    // Первые full_bytes должны быть нулями
    for i in 0..full_bytes { if hash_bytes[i] != 0 { return false; } }

    // Плюс remaining_bits бит в следующем байте
    if remaining_bits > 0 {
        let mask = 0xFF >> remaining_bits;
        if hash_bytes[full_bytes] > mask { return false; }
    }
    true
}

При difficulty=20: первые 2 байта = 0x00, и старшие 4 бита третьего байта = 0. Хеш начинается с 00000... (5 нулевых hex-символов).

Награда за блок (halving)

pub fn calculate_mining_reward(block_index: u64, initial_reward: f64, halving_interval: u64) -> f64 {
    let halvings = block_index / halving_interval;  // сколько раз уже делилось
    initial_reward / 2f64.powi(halvings as i32)     // 1000 → 500 → 250 → ...
}
❓ Что такое Proof-of-Work и почему он нужен?
PoW — это задача, которую сложно решить, но легко проверить. Майнер перебирает значения nonce, пока не найдёт такое, что хеш блока начинается с нужного числа нулевых бит. Это требует вычислительных ресурсов, что защищает сеть от спама и делает фальсификацию блоков экономически невыгодной.
❓ Зачем нужен extra_nonce?
nonce — 64-битное число (максимум ~1.8×10¹⁹). Если перебрать все значения nonce и не найти подходящий хеш, extra_nonce увеличивается — это даёт второй «счётчик», расширяя пространство поиска с 2⁶⁴ до 2¹²⁸.

Получение всех блоков

pub fn get_all_blocks(&self) -> Result<Vec<Block>> {
    let mut blocks: Vec<Block> = self.db
        .scan_prefix(b"block:")   // перебираем все ключи с префиксом "block:"
        .map(|item| serde_json::from_slice(&item?.1))
        .collect()?;
    blocks.sort_by_key(|b| b.index);  // сортируем по порядку
    Ok(blocks)
}

Баланс

pub fn get_balance(&self, address: &str, token_id: &str) -> Result<f64> {
    let key = format!("balance:{}:{}", address, token_id);
    match self.db.get(key.as_bytes())? {
        Some(data) => Ok(String::from_utf8(data.to_vec())?.parse()?),
        None => Ok(0.0),   // если записи нет — баланс 0
    }
}
❓ Почему выбрана sled, а не SQLite?
Sled — встраиваемое key-value хранилище на Rust, не требует установки. Блокчейн сам по себе — структура ключ-значение (номер блока → блок), что хорошо ложится на этот тип БД. Для более сложных запросов подошёл бы SQLite.

Структура Blockchain

pub struct Blockchain {
    db: Arc<Database>,                              // общая БД
    pending_transactions: Arc<Mutex<Vec<Transaction>>>, // пул ожидающих транзакций
    mining_in_progress: Arc<Mutex<bool>>,            // флаг: майнинг идёт

Arc<Mutex<T>> — это обёртка для безопасного разделения данных между несколькими потоками. Arc = счётчик ссылок (несколько владельцев), Mutex = взаимное исключение (только один поток читает/пишет одновременно).

pub async fn add_transaction(&self, transaction: Transaction) -> Result<String> {
    self.validate_transaction(&transaction).await?;
    self.pending_transactions.lock().unwrap().push(transaction.clone());

Валидация транзакции

async fn validate_transaction(&self, tx: &Transaction) -> Result<()> {
    if tx.from != "SYSTEM" {
        if tx.signature.is_empty() { return Err("Missing signature"); }

        let balance = self.db.get_balance(&tx.from, token_id)?;
        if balance < tx.amount + tx.fee { return Err("Insufficient balance"); }
    }
    if let TransactionType::CaseCreate = tx.tx_type {
        if tx.amount < 100.0 { return Err("Requires at least 100 tokens"); }
    }
    Ok(())
}

Майнинг блока

pub async fn mine_block(&self, miner_address: &str) -> Result<Block> {
    // 1. Блокируем флаг — только один майнинг одновременно
    { let mut m = self.mining_in_progress.lock().unwrap();
      if *m { return Err("Mining already in progress"); }
      *m = true; }

    let result = self.mine_block_internal(miner_address).await;
    *self.mining_in_progress.lock().unwrap() = false;
    result
}

async fn mine_block_internal(&self, miner_address: &str) -> Result<Block> {
    // 2. Вычисляем сложность и награду
    let difficulty = BlockUtils::adjust_difficulty(&blocks);
    let reward = BlockUtils::calculate_mining_reward(index, 1000.0, 60480);

    // 3. Проверяем, не превышен ли лимит эмиссии (100M)
    if circulating_supply + reward > self.total_supply { return Err("Cap reached"); }

    // 4. Берём до 100 транзакций из пула
    let txs = pending.drain(..count).collect();

    // 5. Первая транзакция — награда майнеру от SYSTEM
    let reward_tx = Transaction { from: "SYSTEM", to: miner_address, amount: reward, ... };

    // 6. Вызываем PoW: перебор nonce до нахождения нужного хеша
    let new_block = BlockUtils::mine_block(index, transactions, prev_hash, difficulty, ...);

    // 7. Проверяем валидность и сохраняем
    self.process_block_transactions(&new_block).await?;
    self.db.save_block(&new_block)?;
    Ok(new_block)
}

Обработка транзакций блока

async fn process_block_transactions(&self, block: &Block) -> Result<()> {
    for tx in &block.transactions {
        // Списываем у отправителя (amount + fee)
        if tx.from != "SYSTEM" {
            db.set_balance(from, sender_balance - tx.amount - tx.fee, token)?;
        }
        // Зачисляем получателю
        db.set_balance(to, receiver_balance + tx.amount, token)?;

        // Комиссию отправляем на мастер-кошелёк
        if tx.fee > 0.0 {
            db.set_balance(master, master_balance + tx.fee, "REDIPS")?;
        }
    }
    Ok(())
}

❓ Почему reward_tx идёт первой в списке транзакций?
Это стандарт блокчейна — транзакция «coinbase» (награда майнеру) всегда первая в блоке. Она особая: отправитель — SYSTEM (не реальный кошелёк), подпись — просто строка "MINING_REWARD", amount создаётся из воздуха (новые монеты вводятся в оборот).
❓ Что происходит, когда circulating_supply достигает 100M?
Майнинг новых блоков прекращается с ошибкой "Total supply cap reached". Это предел эмиссии, как 21 миллион BTC у Bitcoin. Дальше сеть может существовать только за счёт комиссий с транзакций.

Пример эндпоинта — get_transactions

async fn get_transactions(path: web::Path<String>, db: ...) -> ActixResult<HttpResponse> {
    let address = path.into_inner();
    match db.get_all_blocks() {
        Ok(blocks) => {
            // flat_map: разворачиваем все транзакции из всех блоков в один список
            let mut txs: Vec<Transaction> = blocks
                .into_iter()
                .flat_map(|b| b.transactions)               // все транзакции всех блоков
                .filter(|tx| tx.from == address || tx.to == address) // фильтруем
                .collect();
            txs.reverse();   // новейшие первыми
            Ok(HttpResponse::Ok().json(ApiResponse::ok(txs)))
        }
    }
}

❓ Что такое actix-web и почему Rust?
Actix-web — один из самых быстрых веб-фреймворков в мире (регулярно занимает первые места в TechEmpower). Rust выбран за безопасность памяти без сборщика мусора: компилятор гарантирует отсутствие гонок данных и null-pointer ошибок.
❓ Зачем нужен CORS (Cors::permissive())?
CORS — механизм браузера, ограничивающий запросы с одного домена к другому. Cors::permissive() разрешает запросы с любого источника — удобно для разработки и если API используется фронтендом с другого домена.

9. Ключевые вопросы и ответы

❓ Как обеспечивается целостность блокчейна?
Тремя механизмами: (1) каждый блок содержит хеш предыдущего — изменение любого блока ломает всю последующую цепочку; (2) Merkle root фиксирует все транзакции блока — нельзя добавить/убрать транзакцию незаметно; (3) Proof-of-Work делает пересчёт блоков экономически дорогим.
❓ Что такое halving и зачем он нужен?
Halving — уменьшение награды майнеру вдвое каждые 60480 блоков (~1 год). Это контролирует инфляцию: 1000 → 500 → 250 → ... до достижения лимита 100M монет. Аналогично Bitcoin, где halving происходит каждые 210000 блоков.
❓ Что будет, если два майнера одновременно найдут блок?
В текущей реализации есть флаг mining_in_progress — один узел не может майнить параллельно. В полноценной distributed сети это решается правилом «самой длинной цепочки» — побеждает та ветка, в которую вложено больше вычислений.
❓ Как конкретно проверяется, что у отправителя хватает баланса?
В validate_transaction: читаем из БД баланс адреса tx.from, сравниваем с tx.amount + tx.fee. Если баланса не хватает — возвращаем ошибку и транзакция не попадает в пул. Это называется UTXO-подобная проверка (хотя здесь account-based модель).
❓ Почему блок хешируется дважды (double SHA-256)?
Одинарный SHA-256 уязвим к length-extension attack: зная хеш сообщения M, можно вычислить хеш M+X без знания самого M. Двойной SHA-256 (SHA256(SHA256(data))) защищает от этого. Именно так работает Bitcoin.
❓ Что хранится в поле data транзакции?
Произвольный JSON-объект. Для CaseCreate там будут поля: caseId, price, currency, content (тип и данные контента). Для обычного Transfer — может быть null. Это гибкое расширение без изменения основной структуры.
❓ Как работает мнемоника (seed-фраза)?
По стандарту BIP-39: 128 случайных бит → 12 английских слов. Из слов → 64-байтный seed → SHA256(seed[0..32]) → приватный ключ secp256k1 → публичный ключ → SHA256+RIPEMD160 → адрес. Вся цепочка детерминированная: одна мнемоника всегда даёт один адрес.
❓ Как sled хранит данные на диске?
Sled — это B+-дерево на диске. Данные хранятся в директории (путь из DB_PATH в .env). Ключи и значения — бинарные данные. Мы сериализуем структуры в JSON (через serde_json::to_vec) и сохраняем по строковым ключам типа "block:42".
